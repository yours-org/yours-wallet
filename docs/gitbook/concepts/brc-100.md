# BRC-100

BRC-100 is a BSV standard that defines a stateful wallet interface. A BRC-100 wallet tracks **two things**:

1. The user's keys (BRC-42 derived under various protocols).
2. The user's transaction history (so it can locate outputs to spend).

This is the key shift from earlier injected-provider wallets: the wallet is stateful and knows what the user owns, not just what keys they hold.

## The WalletInterface

The standard exposes a single `WalletInterface` object with categorized methods. Yours Wallet returns this object as `wallet` from `useWallet()`:

```tsx
import { useWallet } from '@1sat/react';
const { wallet } = useWallet();
// wallet implements WalletInterface from @bsv/sdk
```

Method categories (each is a Low-Level docs page):

- **Blockchain queries** — `getHeight`, `getHeaderForHeight`, `getNetwork`, `getVersion`. See [Blockchain Queries](../low-level/blockchain-queries.md).
- **Key derivation** — `getPublicKey`. See [Key Derivation](../low-level/key-derivation.md).
- **Transaction actions** — `createAction`, `signAction`, `abortAction`, `internalizeAction`. See [Transaction Actions](../low-level/transaction-actions.md).
- **Cryptography** — `createSignature`, `verifySignature`, `createHmac`, `encrypt`, `decrypt`. See [Cryptography](../low-level/cryptography.md).
- **Certificates** — `acquireCertificate`, `listCertificates`, `proveCertificate`, `relinquishCertificate`. See [Certificates](../low-level/certificates.md).
- **Discovery** — `discoverByIdentityKey`, `discoverByAttributes`. See [Discovery](../low-level/discovery.md).
- **Key linkage** — `revealCounterpartyKeyLinkage`, `revealSpecificKeyLinkage`. See [Key Linkage](../low-level/key-linkage.md).
- **Output management** — `listOutputs`, `relinquishOutput`. See [Output Management](../low-level/output-management.md).
- **Action history** — `listActions`. See [Action History](../low-level/action-history.md).

## Comparison to legacy injected providers

Older BSV wallets (including legacy Yours Wallet) injected a `window.yours` object with imperative methods like `yours.sendBsv()`. That model:

- Was stateless on the wallet side — the dApp passed UTXOs.
- Lacked a standard interface — every wallet had its own API.
- Had no transaction ancestry concept (no BEEF).

BRC-100 fixes all three. The wallet stores history, the interface is uniform, and BEEF is first-class. See [Legacy Provider Migration](../migration/legacy-provider.md).

## Permission gating

{% hint style="warning" %}
Every BRC-100 method that signs or creates a transaction is permission-gated. The wallet may prompt the user before each call. If they reject, the action returns `error: 'user-rejected'` (or throws).
{% endhint %}

See [Permissions](permissions.md).

## When to use the low-level interface

Most dApps should use the high-level [@1sat/actions](actions-and-context.md) wrappers. Use the raw `wallet.*` methods only when:

- You need custom locking scripts beyond what actions provide.
- You are implementing a protocol not yet wrapped by `@1sat/actions`.
- You need to internalize a payment from an external source via `internalizeAction`.
- You are building certificate-based identity flows.
